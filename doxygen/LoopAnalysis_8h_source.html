86 /// Utility to match a generic reduction given a list of iteration-carried
87 /// arguments, `iterCarriedArgs` and the position of the potential reduction
88 /// argument within the list, `redPos`. If a reduction is matched, returns the
89 /// reduced value and the topologically-sorted list of combiner operations
90 /// involved in the reduction. Otherwise, returns a null value.
91 ///
92 /// The matching algorithm relies on the following invariants, which are subject
93 /// to change:
94 /// 1. The first combiner operation must be a binary operation with the
95 /// iteration-carried value and the reduced value as operands.
96 /// 2. The iteration-carried value and combiner operations must be side
97 /// effect-free, have single result and a single use.
98 /// 3. Combiner operations must be immediately nested in the region op
99 /// performing the reduction.
100 /// 4. Reduction def-use chain must end in a terminator op that yields the
101 /// next iteration/output values in the same order as the iteration-carried
102 /// values in `iterCarriedArgs`.
103 /// 5. `iterCarriedArgs` must contain all the iteration-carried/output values
104 /// of the region op performing the reduction.
105 ///
106 /// This utility is generic enough to detect reductions involving multiple
107 /// combiner operations (disabled for now) across multiple dialects, including
108 /// Linalg, Affine and SCF. For the sake of genericity, it does not return
109 /// specific enum values for the combiner operations since its goal is also
110 /// matching reductions without pre-defined semantics in core MLIR. It's up to
111 /// each client to make sense out of the list of combiner operations. It's also
112 /// up to each client to check for additional invariants on the expected
113 /// reductions not covered by this generic matching.
114 Value matchReduction(ArrayRef<BlockArgument> iterCarriedArgs, unsigned redPos,
115  SmallVectorImpl<Operation *> &combinerOps);
